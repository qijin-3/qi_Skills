# 可信赖 AI 分析的提示模板

即用型提示模板，用于从 AI 获得可靠、可操作的洞察。这些模板可防止常见失败模式（虚构证据、通用洞察、遗漏矛盾）。

## 面试分析的核心提示模板

使用此模板分析用户面试记录。用您的特定上下文替换括号部分。

```
项目上下文
决策：[例如，"探索是否为我们的可穿戴设备添加屏幕"]
约束：[例如，"不得疏远选择我们因为无屏幕的现有用户"]
时间表：[例如，"需要在 2 周内做出决策以用于 Q2 路线图"]

业务目标
我们试图学习的内容：[例如，"添加屏幕是否会吸引新用户而不是疏远现有用户"]
这通知了什么决策：[例如，"优先考虑屏幕开发与其他功能"]
我们需要什么证据：[例如，"明确表明屏幕需求是否代表重大机会或风险"]

产品上下文
产品：[例如，"专注于恢复和睡眠跟踪的无屏幕健身可穿戴设备"]
关键差异点：[例如，"无屏幕意味着更少的干扰、更好的电池、极简设计"]
竞争格局：[例如，"与 Apple Watch、Garmin、Oura Ring 竞争"]
关键权衡：[例如，"添加屏幕会使我们更像竞争对手"]

参与者概述
总参与者：[N]
当前用户：[N] - [简要描述，例如，"长期用户，重视简单性"]
已流失用户：[N] - [简要描述，例如，"转向 Garmin/想要屏幕"]
潜在用户：[N] - [简要描述，例如，"当前使用竞争对手产品"]

面试记录
[在此粘贴记录，按参与者 ID 清晰分隔]

分析要求

1. 引言选择规则
从思想开始的地方开始，直到完全表达
包含推理，而不仅仅是结论
保留模糊限制词 - 它们表示不确定性
包含情绪语言（如果有）
使用参与者 ID 和大约时间戳引用 [P02 ~14:30]
不要组合面试中不同部分的陈述
如果引言超过 3 句话，将其拆分为单独的引言

2. 证据锚定
对于每个洞察，提供具体的参与者引用
注意提到此内容的参与者比例（例如，"3/10 参与者提到"）
当证据薄弱时标记（例如，"基于单次提及，可能不具有代表性"）

3. 矛盾检查
寻找并报告矛盾的陈述
注意参与者在面试中何时改变主意
报告人们所说的与他们行为之间的张力
区分不同用户细分想要的东西

4. 风险标记
识别分析中的假设
标记与主要主题相矛盾的证据
注意可能重要的边缘情况或异常值
如果置信度低，请说明

5. 细微差别优于简化
不要过度简化复杂的模式
存在混乱时报告混乱
区分"一些用户想要 X"与"大多数用户想要 X"
不要将表面级请求解释为字面功能要求

输出格式

按此结构提供分析：

## 执行摘要
[2-3 段关键发现概述，包括用户细分和张力]

## 按主题的洞察
[对于每个主题：]
### [主题名称]
- **用户所说**：[摘要与逐字引言和引用]
- **谁说的**：[哪些用户细分，多少参与者]
- **证据强度**：[强/混合/弱 - 解释原因]
- **矛盾**：[任何矛盾的证据或张力]
- **业务影响**：[这对决策意味着什么]
- **置信度水平**：[高/中/低]

## 矛盾和张力
[报告参与者之间的矛盾、面试内部的矛盾，或所述偏好与行为之间的矛盾]

## 用户细分差异
[如果不同细分想要不同的东西，分别报告]

## 风险和假设
[标记分析中的任何假设、弱证据或风险]

## 推荐
[基于证据，您会推荐什么？什么额外研究将澄清剩余的不确定性？]
```

## 引言验证提示词

收到 AI 分析后，始终运行此验证提示词以捕获虚构或释义的引言：

```
引言验证

对于上述分析中的每条引言：

1. 确认引言在源记录中逐字存在
2. 如果引言是接近的释义但不确切，标记它并提供实际措辞
3. 如果无法找到引言，标记为未找到

对于每条引言的输出格式：

引言：[分析中的引言]
状态：已验证 / 释义 / 未找到
如果释义：实际措辞：[他们实际说的内容]
位置：[参与者 ID，时间戳或行号]
```

## 调查分析提示模板

用于分析调查数据（来自 SurveyMonkey、Qualtrics 等的 CSV 导出）：

```
项目上下文
决策：[此研究通知的特定决策]
约束：[决策的任何约束]
业务目标：[您试图学习的内容]

产品上下文
产品：[简要产品描述]
关键差异点：[您的不同之处]
竞争格局：[您与谁竞争]

调查上下文
调查类型：[例如，"流失调查"、"NPS 调查"、"功能兴趣调查"]
总响应：[N]
时间段：[调查进行的时间]

列规范
客户声音列：[列出包含实际客户响应的列]
要忽略的元数据：[列出要忽略的列 - 时间戳、内部标签等]
量表定义：[如果使用评分量表，定义它们：例如，"1-5 量表，其中 5 最好"]

模糊性指导
[澄清模糊术语。示例：]
- 当用户说"不值得"时，他们可能意味着：太贵、不够有价值、错误功能等 - 请区分这些
- 当用户说"很难使用"时，指定他们的意思是：令人困惑的 UI、技术问题、学习曲线等
[添加您的特定模糊术语和解释]

分析要求

1. 不要将不同的含义归入通用主题
2. 要区分相似响应的不同原因
3. 当响应过于模糊无法分类时报告
4. 不要发明解释 - 将不清晰的响应标记为"不清晰"
5. 报告每个主题的响应比例
6. 标记主题与我们做出的决策没有直接关系时

输出格式

## 响应主题
[对于每个不同的主题：]
### [主题名称]
- **代表性响应**：[逐字示例，保持它们区分]
- **响应计数**：[N 响应，总计的 X%]
- **解释**：[这些响应意味着什么，如果有的话]
- **与决策的相关性**：[这与我们的特定决策有何关系]
- **清晰度**：[清晰/混合/模糊 - 解释响应是否模糊]

## 不清晰或模糊的响应
[报告无法分类的响应 - 这些也是重要的信号]

## 矛盾
[如果响应相互矛盾或与其他数据矛盾，请报告]

## 推荐
[这对我们的决策意味着什么？什么额外问题会有帮助？]
```

## 快速参考：上下文加载清单

在分析之前，确保您的提示词包括：

- [ ] **项目上下文**：我们正在做出什么决策？有什么约束？
- [ ] **业务目标**：我们试图学习什么？我们需要什么证据？
- [ ] **产品上下文**：我们在构建什么？我们与谁竞争？我们的权衡是什么？
- [ ] **参与者概述**：这些人是谁？（当前用户、已流失、潜在用户？）
- [ ] **列规范**（仅调查）：哪些列是客户声音与元数据？

## 最佳结果提示

1. **具体说明决策** - "探索是否添加屏幕"比"做研究"更好
2. **定义约束** - 帮助 AI 理解什么解决方案是排除的
3. **指定用户细分** - "已流失的 Garmin 切换者"与"用户"告诉 AI 要寻找什么
4. **始终验证引言** - 在使用洞察之前运行验证提示词
5. **要求细微差别** - 明确要求 AI 报告矛盾和张力
6. **标记不确定性** - 要求 AI 报告置信度水平和弱证据
7. **不要过度简化** - 真实研究是混乱的；要求 AI 报告复杂性

## 避免的常见提示错误

- ❌ "给我有力的引言（≤12字）" - 触发弗兰肯斯坦引言
- ❌ "每条引言最多 100 字" - 导致引言拼接
- ❌ "总结用户所说的" - 失去细微差别和确切措辞
- ❌ 模糊的上下文，如"做用户研究" - 导致通用洞察
- ❌ 在提示词中提到特定问题（"定价问题"）- 使分析产生偏见
- ❌ 要求"代表性引言"而不定义这意味着什么

## 何时使用此参考

在以下情况下阅读本文档：
- 准备分析面试记录或调查响应
- 为 AI 驱动的用户研究设计提示词
- 培训团队有效的 AI 研究提示
- 诊断为什么 AI 分析感觉通用或不可信
