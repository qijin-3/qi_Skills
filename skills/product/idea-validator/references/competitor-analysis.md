# 竞品分析方法论 · idea-validator 专用版

## 核心理念

不是找「谁做得更好」，而是找「市场格局中哪个象限是空的，且有用户需求在那个方向」。

独立开发者的竞争策略不是正面攻打行业龙头，而是找到「大公司不屑做、小团队没做好」的有效空白，用最小资源验证最快路径。

---

## 竞品识别策略

### 第一层：从 Step 2 用户声音中提取

Step 2 的搜索结果会天然暴露竞品名称：
- 用户在抱怨某个产品时提到的名字
- 推荐帖子里被高频提及的工具
- 用户说「我从 X 换到 Y」时出现的产品名

**这是最可靠的竞品来源**：能出现在用户抱怨和推荐讨论里的，是真实的竞争对手。

### 第二层：补充搜索

用以下搜索补全竞品视野：
```
best [产品类别] tools / apps
[产品类别] alternatives 2024 / 2025
[产品类别] indie / solo        ← 专门找独立开发者做的同类产品
top [产品类别] reddit           ← 看 Reddit 推荐帖
```

通过 iTunes API 找应用商店竞品：
```bash
curl "https://itunes.apple.com/search?term=[产品类别]&entity=software&limit=10&country=us"
```

通过 Algolia HN 找技术圈竞品讨论：
```bash
curl "https://hn.algolia.com/api/v1/search?query=[产品类别]&tags=story"
```

### 竞品数量要求

- **目标**：3–6 个主要竞品
- **少于 3 个**：标注「竞品覆盖不足，定位矩阵置信度低，建议手动补充」
- **超过 6 个**：优先选最相关（用户提到频率高）的 6 个，其余简要列出

## 竞品数据收集字段

对每个竞品收集以下字段：

| 字段 | 要求 | 来源 |
|------|------|------|
| 产品名 + 官网 | 附链接 | 官网 / App Store |
| 定价模型 | 免费 / 订阅 / 买断 / Freemium，写明具体价格 | 官网定价页 |
| 核心定位 | 他们自己说的是什么，一句话 | 官网首屏 / App Store 描述 |
| 目标用户 | 个人 / 小团队 / 企业 | 官网 + 用户评论 |
| 用户高频抱怨 | 来自 App Store / Google Play / G2 的 1–2 星评论，提取 Top 3 | Step 2 搜索结果 |
| 社区切换原因 | 用户在 Reddit / HN / X 提到的「为什么离开」 | Step 2 搜索结果 |

**获取方式快速参考**：
```bash
# 官网内容（用 Jina Reader）
curl "https://r.jina.ai/https://[竞品官网]"

# App Store 评论（两步法）
curl "https://itunes.apple.com/search?term=[竞品名]&entity=software&limit=3"
# 取 trackId 后：
curl "https://r.jina.ai/https://apps.apple.com/us/app/id[APP_ID]"

# Google Play 评论
curl "https://r.jina.ai/https://play.google.com/store/apps/details?id=[包名]"
```

## 竞品定位矩阵构建

### 坐标轴选择原则

**不要用固定轴**。根据实际竞品分布动态决定，选择**最能区分当前这批竞品**的两个维度。

**选择方法**：
1. 列出所有竞品的定位描述
2. 找出这些竞品在哪些维度上差异最大
3. 选差异最大的两个维度作为坐标轴

**常用维度示例**（不是强制选项）：
- 目标用户规模：个人 vs 团队 / 企业
- 功能复杂度：轻量 vs 专业 / 全功能
- 价格区间：免费 / $10 vs $50 / $100+
- 自动化程度：手动操作 vs 高度自动化
- 内容深度：通用 vs 垂直细分领域
- 部署方式：云端 vs 本地 / 自托管

**坐标轴选择说明**：在报告中说明「选择这两个维度是因为...」，让读者理解矩阵的构建逻辑。

### 矩阵填充

将所有竞品标注在矩阵象限中，识别：
1. **竞争最激烈的象限**：多个竞品聚集
2. **明显空白的象限**：无成熟竞品覆盖
3. **目标 idea 的潜在落点**：建议落在哪个象限

## 空白区有效性判断

### 「有效空白」必须同时满足两个条件

**条件 1**：矩阵中没有成熟竞品覆盖该象限
**条件 2**：Step 2 的用户声音中有证据显示这个方向有需求

**仅满足条件 1 不够**——没有竞品不等于有市场，可能只是因为市场不存在。

### 常见误判场景（这不是有效空白）

❌ **「这个功能没有人做」**：功能缺失不等于市场空白，可能只是需求太小，做了没人用。

❌ **「我的定价比竞品低 50%」**：价格差异不是定位矩阵意义上的空白，只有当低价对应的是不同用户群或不同使用场景时才算。

❌ **「我针对中国市场，竞品都是英文的」**：本地化是差异化，但只有当你找到了有真实用户声音支持的中文需求时才算有效空白。

❌ **「这个象限只有一个小团队在做，他们做得不好」**：需要验证「做得不好」是用户也这么认为（有抱怨证据），还是只是你的主观判断。

### 有效空白的表述模板

```
识别到有效空白：[象限描述]
空白证据：矩阵中无成熟竞品覆盖此象限
需求证据：Step 2 中有 X 条用户声音显示 [具体需求方向]，
          来自 [平台名]，可信度 [高/中]
```

## 独立开发视角的特殊考量

### 大公司不屑做的空间

大公司有几类事情倾向于不做：
- **客单价太低**（$5–$15/月的个人工具）
- **用户量级太小**（面向某个垂直细分的几万用户）
- **品牌风险**（处理敏感数据或需要信任关系的场景）
- **技术整合太复杂**（需要与多个小众工具打通）

这些恰好是独立开发者的机会区间。

### 小团队没做好的迹象

通过竞品评论找到以下信号，说明该方向有待改进：
- 用户抱怨某个功能「一直 broken」（小团队资源不足维护）
- 用户说「开发者不回复 issue / feedback」（没有 PMF 后失去动力）
- 评论区出现「这个产品已经停止更新了」
- App Store 评论区出现「已经付钱但 bug 几个月没修」

这些信号 = 市场有需求，但现有供给质量低，是切入机会。

### Step 3 完成输出格式

```
主要竞品（3–6个）：
- [竞品1]：[定价] / [核心定位] / Top 1 抱怨
- [竞品2]：...

竞争最密集区域：[象限描述，说明为什么密集]

识别到的有效空白：[空白象限描述]
  - 空白证据：[无竞品的说明]
  - 需求证据：[Step 2 对应用户声音条数和平台]
  （或「未识别到有效空白」+ 原因）

差异化切入建议：[具体的差异化方向，不是泛泛的「做得更好」]
```
