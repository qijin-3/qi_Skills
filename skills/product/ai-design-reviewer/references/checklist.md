基于 8 种核心用户意图框架的设计评审清单。

---

## 第一步：识别核心意图

**你的 AI 功能主要支持哪个（或哪些）意图？**

- [ ] **Know/Learn** - "我想理解这个"
- [ ] **Create** - "我想创建或改变这个"
- [ ] **Delegate** - "帮我做这个"
- [ ] **Oversee** - "让我介入并保持控制"
- [ ] **Monitor** - "保持我了解和更新"
- [ ] **Find/Explore** - "帮我找到和比较选项"
- [ ] **Play** - "娱乐我"
- [ ] **Connect** - "被倾听；陪伴"

## 第五步：关键能力检查（分意图）

### Know/Learn - 知识获取
- [ ] ✅ 先给出答案，再解释
- [ ] ✅ 显示工作过程（可点击的源链接）
- [ ] ✅ 后续跟进建议（显示数据、比较、深入探讨）
- [ ] ❌ 避免倾倒未结构化的长文
- [ ] ❌ 避免盲目自信地犯错

### Create - 内容创作
- [ ] ✅ 始终提供起点（模板/示例）
- [ ] ✅ 除了提示词外，提供控件（语气/长度/风格）
- [ ] ✅ 支持基于区域的编辑（只重新生成选中部分）
- [ ] ✅ 版本控制 + 差异对比
- [ ] ❌ 不要强迫"提示词体操"

### Delegate - 任务委托
- [ ] ✅ 预览计划（纯语言步骤、工具调用、护栏）
- [ ] ✅ 提供模拟预飞
- [ ] ✅ 实时执行可见性（运行面板、活动中心）
- [ ] ❌ 不要静默执行或不可逆操作
- [ ] ❌ 不要将"操作"伪装成"聊天回复"

### Oversee - 监督审查
- [ ] ✅ 解释为什么需要审查（推理卡片）
- [ ] ✅ 一键操作（批准/拒绝/编辑）
- [ ] ✅ 显示证据，不仅仅是声明
- [ ] ❌ 不要倾倒无上下文的差异对比

### Monitor - 监控更新
- [ ] ✅ 结构化摘要
- [ ] ✅ 可调频率（摘要 vs 警报）
- [ ] ✅ 解释"为什么显示这个"
- [ ] ❌ 不要做 AI 垃圾邮件大炮

### Find/Explore - 搜索探索
- [ ] ✅ 支持多模态/混乱输入
- [ ] ✅ 解释推荐理由
- [ ] ✅ 支持保存和对比
- [ ] ❌ 不要做黑盒推荐

### Play - 娱乐
- [ ] ✅ 预设和模板（移除提示词工作）
- [ ] ✅ 结构化会话（有开始、循环和结束）
- [ ] ✅ 可收集和分享的时刻
- [ ] ❌ 不要纯时间陷阱（暗模式）

### Connect - 连接陪伴
- [ ] ✅ 明确的关系契约（朋友 vs 教练）
- [ ] ✅ 情感记忆和回溯
- [ ] ✅ 安全护栏（年龄模式、内容边界）
- [ ] ❌ 不要伪装成治疗师

## 第六步：安全性和可逆性检查

**对于所有可变操作：**
- [ ] ✅ 提供 Plan Preview（计划预览）
- [ ] ✅ 提供 Dry Run（模拟运行）
- [ ] ✅ 提供 Undo/Revert（撤销/还原）
- [ ] ✅ 提供 Audit Trail（审计日志）

---

## 第七步：反模式自检

**检查你是否犯了这些错误：**

- [ ] ❌ **强迫提示词体操** - 常见变换应该是按钮
- [ ] ❌ **没有安全网覆盖内容**
- [ ] ❌ **无预览的静默更改**
- [ ] ❌ **静默或不可逆地执行**
- [ ] ❌ **将"操作"伪装成"聊天回复"**
- [ ] ❌ **过度承诺"通用代理"能力**
- [ ] ❌ **倾倒无结构化的论文**
- [ ] ❌ **盲目自信地犯错**
- [ ] ❌ **过度解释** - 用户问快捷键，不要给历史课
- [ ] ❌ **黑盒回复** - 没有来源和推理过程
- [ ] ❌ **一次性生成** - 无法迭代和局部修改
- [ ] ❌ **通知噪音** - 过度打扰用户
