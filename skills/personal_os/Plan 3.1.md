# Personal OS 3.1 方案文档

> 版本：3.1（基于 3.0 升级）
> 升级重点：① 补全北极星生成与维护机制 ② 月/周目标共创加入 AI 守门人协议
> 配置源：**唯一** `{PERSONAL_OS_ROOT}/references/config.md`

---

## 〇、一页速览

**一句话**：你只管"记任务、定方向、写碎片、给反馈"，Hermes 用飞书 Task / Base 管运营数据，长文本地 md 编辑，Wiki 只同步 About Me 正文快照，并长期守护你的方向不漂移。

```
你（飞书发消息）           Hermes（Mac Mini 上的 Agent）           沉淀
─────────────────         ─────────────────────────────         ──────────────
记任务      ──────────▶   建任务 + 分清单 + 设重要性        ──▶  飞书 Task
写碎片日记  ──────────▶   原文存档 + 每晚拼装               ──▶  本地 diary.md
（不操作）   每日早报  ──▶  排今天要做 1–3 条 + 写评论 + 本地快照   ──▶  state/daily_snapshot
（不操作）   每日晚报  ──▶  统计完成 + 偏差预警 + 拼日记 + 写日志表  ──▶  日志表 + 本地 diary.md
和 AI 共创   每周/每月 ──▶  守门审查 + 复盘上期 + 编排下期    ──▶  本地 md + 飞书目标表
给反馈      （被追问时）  调整任务/分组（大方向默认不动）     ──▶  分组审计表
关于我更新  （月频提案）  先写当月更新记录 → 改本地 → 同步飞书     ──▶  本地 md + Wiki「关于我」
```

**核心信念**：复利来自方向稳定。系统的首要职责不是"帮你做更多"，而是"帮你少做、做对、长期不换方向"。

> **3.1 新增**：AI 在目标共创中的默认立场从「优化者」变为「守门人」。共创不是帮你把想法写漂亮，而是先质疑、再打磨、最后确认。

## 一、方法论基础

### 1.1 四条第一性原则

| # | 原则 | 出处 | 一句话 |
|---|------|------|--------|
| P1 | **复利来自方向稳定** | 娜瓦尔：与长期的人玩长期的游戏 | 频繁换方向 = 复利清零；目标一旦定下，默认不变 |
| P2 | **把精力压在高杠杆上** | 娜瓦尔：代码与媒体是零边际成本的杠杆 | 写作=媒体杠杆，成长/项目=产品杠杆；优先喂养它们 |
| P3 | **外化大脑，换取内心平静** | 生产力：把一切写下来 | 大脑用于思考而非存储；记任务/记碎片即清空大脑 |
| P4 | **按能量调度 + 用数据寻路** | 能量管理 + 人生设计 | 不只排时间，还要识别什么让你"活过来"，并据此校准 |

### 1.2 设计决策 ← 原则的映射

| 设计 | 对应原则 | 理由 |
|------|----------|------|
| 北极星/月目标默认不可变（§7.4） | P1 | 守护复利，抵抗"优化即拖延" |
| 五清单按杠杆分类 | P2 | 让高杠杆产出显性化、易被优先调度 |
| 今天要做 ≤3 条 | P2 P3 | 聚焦少数关键，拒绝是默认 |
| quick-task 一句话建任务 | P3 | 捕获摩擦越低，外化越彻底 |
| 日记原文存档（零改写） | P4 | 原始语料是寻路的唯一可信数据 |
| 分组审计表（计划 vs 实际的博弈） | P1 P4 | 量化偏差，识别动机与心理模式 |
| AI 守门人协议（§7.5）**【3.1 新增】** | P1 | 用结构化质疑替代顺从式共创 |

### 1.3 计划稳定性铁律（系统的脊柱）

> AI 的默认立场是**守护者**，不是**优化者**。

1. **北极星 / 年度方向**：除非人生级变故，**全年不改**。
2. **月目标**：每月 1 号共创定稿后，月内默认不改；新机会先进「未安排」，下月再议。
3. **周主线**：周日定稿后，周内默认不改；临时事项默认顺延，不挤主线。
4. **唯一例外通道**：用户明确说出"这是特殊情况" → AI 必须先**复述放弃的代价**（中断了哪条复利），用户确认后才动。
5. AI 检测到偏差时，**先预警、再对话、后调整**，永远不自作主张动月/周大方向。

## 二、角色分工

### 2.1 用户只做 4 件事

1. **记任务** — 飞书随手发一句话
2. **定方向** — 每年年初制定北极星；每周日 / 每月 1 号和 Hermes **共创**目标（接受守门审查，不是 AI 单方面出稿）
3. **写碎片日记** — 白天想到什么发什么
4. **给反馈** — 被晚报 / 偏差预警追问时，说明为什么没做或要不要调整

### 2.2 AI 做的事

#### A. 实时响应（事件驱动）

| 动作 | 技能 | 说明 |
|------|------|------|
| 增加任务 | `quick-task` | 分清单 + 设重要性 + 负责人；不打 Tag |
| 捕获日记碎片 | `diary-capture` | 原文存档，不改写 |
| 移动「我负责的」分组 | （早晚报/指令内） | 仅在 5 个执行分组间移动，并写审计 |
| 临时事项研判 | Hermes 对话 | 你说"我想加 X，合理吗" → 判断代价、建议放哪 |

#### B. 定时编排（cron 驱动）

| 频率 | 技能 | 职责 |
|------|------|------|
| 每日 09:00 | `daily-briefing` | 排今天要做 1–3 条 + 写评论 + 偏差扫描 |
| 每日 22:00 | `evening-review` | 统计完成 + 完成率预警 + 拼日记 |
| 每周日 20:00 | `weekly-goal` | 上周分析 + 守门审查 + 下周编排 |
| 每月 1 号 10:00 | `monthly-goal` | 月计划守门共创 + 拆周 |
| 每年 1 月 1 号 | `annual-review` **【3.1 新增】** | 年度复盘 + 北极星制定/校验 |

### 2.3 AI 明确不做

创建后改标题/描述/重要性；改清单内分组；替用户完成任务；**改写日记原文**；**擅自变更月/周/年大方向**；跳过守门流程直接写入目标。

### 2.4 记任务（quick-task）简化规则

用户说一句话 → AI 自动完成：选清单、设重要性（高/中/低）、设负责人、落入「未安排」。

| 用户意图 | 清单 | 杠杆属性 |
|----------|------|----------|
| 想法 / 灵感 / 还没决定做 | 💡 Idea | 原料 |
| 写作 / 自媒体 / 文章 | ✍️ Writing | 媒体杠杆 |
| 技能 / 系统 / 学习效率 / 工具 | ♻️ Growing | 复利学习 |
| 长期项目 / 多步骤交付 | 📦 Projects | 产品杠杆 |
| 日常 / 重复 / 杂务 | 🔁 Routine | 维持成本 |

重要性默认**中**；"紧急/重要/优先" → **高**；"以后再说/低优" → **低**。

## 三、配置管理（单一数据源）

所有飞书 token、GUID、URL 只写在 `{PERSONAL_OS_ROOT}/references/config.md`。本地路径模板写在同一文件的 `PATH_*` 变量中（**相对路径**，随 Synology 同步，禁止写机器专属绝对路径）。技能 / cron 只写变量名，不写具体值。

### 3.1 多机路径（Synology 同步）

各 Mac 的 Synology 挂载路径不同，技能通过 `PERSONAL_OS_ROOT` 占位符访问文件，执行前按 `{PERSONAL_OS_ROOT}/references/path-resolution.md` 解析。

**推荐**：每台机器一次性创建统一入口：

```bash
# 主 Mac
ln -sfn "/Users/jin/SynologyDrive/Sync/OS/Personal_OS" ~/Personal_OS

# 另一台 Mac
ln -sfn "/Users/mac_jin/SynologyDrive/SynologyDrive/Sync/OS/Personal_OS" ~/Personal_OS
```

解析顺序：`$PERSONAL_OS_ROOT` 环境变量 → `~/Personal_OS` → 常见 Synology 路径自动探测。

| 模式 | 判断逻辑 |

## 六、每日循环

### 6.1 早报 09:00（`daily-briefing`）

1. 读 `TABLE_GOALS`（本月/本周进行中）+ `MY_*` 各分组任务
2. **偏差扫描**：新任务高重要性或与目标冲突 → 预警
3. 按"高杠杆优先 + ≤3 条"挑今天要做，移动写审计
4. 每条附评论（为什么是今天）
5. 保存 `state/daily_snapshot_{date}.json`（含晨间 `MY_TODAY` 计划）+ 飞书消息摘要

### 6.2 晚报 22:00（`evening-review`）

1. 读当天 `completed_at` 任务；从快照读晨间计划 → 算完成率
2. **完成率预警**：连续 2 天完成 < 1/3 → 发起对话
3. 未完成项 → 评论 + 移分组 + 写审计
4. `diary-assembler` 拼日记 → 写入本地 `diary.md`
5. **统一 upsert** `TABLE_LOGS`（今日计划 + 完成任务 + 完成度 + 主线）+ 快照 diff

### 6.3 随时

- 记任务 → `quick-task`（落 `MY_UNPLANNED`）
- 记日记 → `diary-capture`（原文入本地 pending）

## 七、目标体系：制定、守门与复盘

### 7.1 目标层级

```
北极星（全年，P1 默认不变）
   └─ 年度目标（≤3 条，年内不变）
        └─ 月目标（≤3 条，月内不变）
             └─ 周主线（1 条/周，周内不变）
                  └─ 今天要做（≤3 条/日，可微调）
```

### 7.2 北极星：生成与维护 **【3.1 新增核心章节】**

> 北极星是整个系统的最高约束层。月目标、周主线所有决策都以「是否推进北极星」为终极判断标准。

#### 7.2.1 首次生成（或年度重制）

北极星在**每年 1 月 1 号**由 `annual-review` 技能触发，与用户共创生成。流程：

```
Step 1：AI 展示上年客观数据
  - 各清单完成量与比例
  - 月度完成率趋势（12 个月折线）
  - 目标表：北极星三支柱各完成度
  - 日记能量信号：哪些月份活跃/倦怠

Step 2：AI 提出方向检验问题（用户必须回答，不能跳过）
  - "去年花时间最多的是什么？它今年还值得继续吗？"
  - "去年哪件事给你带来了最强的心流体验？"
  - "如果只保留一个方向，你会选什么？"

Step 3：AI 根据 About Me + 上年数据 + 用户回答，起草北极星草案
  - 核心方向（1–2 句话）
  - 三个战略支柱（≤3 个）
  - 全年刻意放弃清单
  - 年底衡量标准

Step 4：AI 过北极星质量门（必须全部通过才能写入）
  ① 聚焦检查：三支柱加起来一个人能做到吗？有没有明显的精力冲突？
  ② 可测量检查：年底衡量标准能在 30 秒内判断是否达到？
  ③ 杠杆检查：是否至少有一个支柱属于媒体或产品杠杆？
  ④ 放弃检查：「刻意放弃」清单是否足够狠？

Step 5：用户确认 → 写入 content/north-star.md + TABLE_GOALS（层级=北极星）
```

#### 7.2.2 月度对齐检查（内嵌于 monthly-goal）

每次月度共创，AI 必须在 Step 1 展示：

```
本月目标草案 → 对应北极星支柱：[X]
上月该支柱推进情况：[完成度]
本月若不做这个支柱，年底衡量标准能达到吗？
```

若某个月的目标完全不覆盖任何一个北极星支柱，AI **必须提出质疑**，不能直接写入。

#### 7.2.3 中途修订（唯一例外通道）

北极星修订门槛极高，需同时满足：

- 用户主动提出（AI 不主动建议修订北极星）
- 用户明确说出"我想修改北极星"
- AI 先复述：放弃当前方向会中断哪条复利，已投入多少时间/产出
- 用户二次确认后，才写入修订记录并更新文件

平时的目标建议调整、月内计划微调，不属于北极星修订。

### 7.3 年度复盘与北极星校验（每年 1 月 1 号）

```
1. Hermes 拉取全年数据：完成率趋势、各支柱完成度、日记能量高峰
2. 展示年度复盘摘要
3. 北极星检验：去年三支柱完成度如何？方向还对吗？
4. 共创新一年北极星草案（过质量门）
5. 创建年度目标（≤3 条，对应三支柱）
6. 写入 content/YYYY/annual-review.md + 更新 north-star.md
```

### 7.4 月度共创 + 拆周（每月 1 号，约 20 分钟）

```
Step 1：AI 展示上月客观数据
  - 任务完成数与各清单分布
  - 今天要做平均完成率
  - 分组博弈 Top 模式（ai_in_user_out / stale_*）
  - 北极星各支柱本月推进情况
  - 日记能量信号（上月有无倦怠/高效的规律）

Step 2：AI 提问有效工作日
  "本月有多少有效工作日？（排除旅行、社交密集周）
   按你实际每天约 X 小时深度时间，可用总时间约 Y 小时。"

Step 3：过 AI 守门四关（见 §7.5）
  用户提议目标 → AI 逐条审查 → 不通过必须挑战

Step 4：对话打磨确认
  守门通过后，用户最终确认 ≤3 条目标

Step 5：拆周主线 + 写入
  确认 → 写本地 monthly-plan.md + 四个 md 文件 + 写目标表

Step 6：为上月月度目标打分
  AI 提议分数 → 用户确认 → 写目标完成度（0–1）+ 更新状态
```

### 7.5 AI 守门人协议 **【3.1 新增核心章节】**

> **原则**：AI 在目标共创中的默认立场是「质疑者」，不是「优化者」。共创的目的不是把你的想法写漂亮，而是帮你发现想法的漏洞，再一起打磨成能执行的目标。

#### 守门四关（月度目标必须全部通过）

**关卡 1：可验证性检查**

每条目标必须有明确的验收标准，能在 30 秒内判断达到与否。

> AI 挑战话术：
> "「提升 AI 产品能力」这个目标，月底怎么判断做到了？如果没有具体标准，这条还不够清晰，我们来打磨一下。"

不合格 → AI 拒绝写入，追问直到验收标准明确。

**关卡 2：负荷合理性检查**

基于有效工作日 × 实际深度时间，计算可用总时间，每条目标预估耗时，叠加是否超过可用时间的 70%。

70% 上限原因：你存在注意力分散、项目切换消耗、晚睡晚起等已知问题（见 About Me），满负荷计划必然超支。

> AI 挑战话术：
> "你本月有效工作日 N 天，按实际节奏可用深度时间约 X 小时。
>  这 3 条目标加起来预估需要 Y 小时，超出了 Z 小时。
>  建议删掉哪条，或者把哪条降级到「刻意不做」？"

超出 70% → AI 必须要求删减，不能帮你"挤一挤"通过。

**关卡 3：历史完成率对抗**

如果上月完成率 < 60%，本月计划不得比上月更重（条数 × 难度综合判断）。

> AI 挑战话术：
> "上月完成率是 X%，本月计划比上月更重（从 A 条增加到 B 条）。
>  上月哪里卡住了？本月有什么具体的不同让你觉得能完成更多？
>  如果没有新的变量，建议先从 B 条降回 A 条，把质量做好。"

用户必须说明"本月的不同变量"才能维持重于上月的计划；否则 AI 坚持建议降级。

**关卡 4：北极星对齐检查**

每条目标必须说明贡献哪个北极星支柱。若有目标与北极星无明确关联，AI 必须 flag。

> AI 挑战话术：
> "「研究新的设计工具」这条目标跟你北极星的三个支柱（写作杠杆/产品杠杆/能力复利）的关系我看不清楚。
>  它是支撑哪个支柱的？如果本月只能完成 1 条，你会选它吗？"

无法说明贡献路径 → AI 建议移入「刻意不做」或降级至「未安排」。

#### 守门后的记录义务

即使用户坚持某条目标通过了守门（AI 挑战后用户给出了解释），AI 也必须在 `monthly-plan.md` 中写入：

```markdown
## 守门备注
- 目标「X」：用户确认接受[负荷超标/历史风险/对齐模糊]风险，理由：[用户说的]
```

这条备注在下月复盘时会被回溯引用。

#### 守门话术原则

- **问句优先于否定句**：用"这条目标是不是可以更具体？"而不是"这条目标不够好"
- **数据先于判断**："上月完成率 40%"比"你上月做得不好"更有说服力
- **给出选项而非命令**：永远提供 2 个以上的路径供用户选择
- **尊重用户最终决定**：守门通过不了时，用户若坚持 → 写入备注，继续执行，不阻断流程

### 7.6 周度复盘 + 编排（每周日，约 10 分钟）

```
Step 1：分析上周
  - 完成度：主线完成情况 + 今天要做平均完成率
  - 偏离：哪条没推进、为什么
  - 目标进展：对齐本月目标的百分比
  - 北极星支柱本周推进情况【3.1 新增】
  - 可提升点：≤2 条，来自分组博弈 + 日记能量信号

Step 2：动量检测【3.1 新增】
  若某个月目标连续 2 周完全无推进：
  "本月目标「X」到目前为止连续 N 天没有相关任务完成。
   下周主线建议指向这里。如果你有其他考虑，请说明。"
  → AI 不能因用户没提就跳过这个目标

Step 3：下周时间核验【3.1 新增】
  "下周有没有特殊安排（出行/会议密集）？"
  → 基于实际可用时间给主线，不默认满周可用

Step 4：提议下周 1 条主线
  对齐本月目标 + 进度最慢支柱 + 实际可用时间

Step 5：确认 + 为上周目标打分 + 写入
  用户确认分数 → 写 weekly-review.md →
  **归档上周** TABLE_WEEKLY（完成度 + 状态 + 总结，逐条 upsert）
  ※ 创建下周新目标由 weekly-plan 负责，不在周复盘内
```

### 7.7 偏差与计划调整机制

> 原则：**先预警、再对话、后调整；大方向默认不动。**

**三个检测时机：**

| 时机 | 触发条件 | AI 动作 |
|------|----------|---------|
| 每日早报 | 月/周/今天分组冒出新任务且高重要性或与目标冲突 | 预警："这条是新冒出来的，纳入本周还是放未安排？它会挤掉 X" |
| 随时（手动） | 你说"我想加 X，合理吗" | 研判是否影响目标，给出代价与建议 |
| 每日晚报 | 连续 2 天完成率 < 1/3 | 对话："是任务太多，还是方向不对？" |

**调整分级：**

| 级别 | 范围 | 门槛 |
|------|------|------|
| 微调 | 今日任务换序/顺延 | AI 自主，记审计 |
| 中调 | 本周主线更换 | 必须对话确认 |
| 大调 | 月目标变更 | 你明确说"特殊情况" → AI 先复述放弃的复利代价 → 你再确认 |
| 极大调 | 北极星 / 年度方向变更 | 两次确认 + 写修订记录（见 §7.2.3） |

## 八、日记：捕获与拼装

### 8.1 设计原则

**你的文字是原文资产。** AI 只按时间排好、分段，**不得**总结、改语气、替你表达。

| 允许 | 禁止 |
|------|------|
| 按时间排序、空行分段 | 改写句子、加 AI 观点/情绪解读 |
| 保留每条原文 | 把多条合并成"更好读"的一句 |
| 去掉完全重复的连续 duplicate | 删减你觉得"不重要"的碎片 |

### 8.2 diary-capture（实时）

**触发**（满足任一即视为日记）：显式标记（`#日记`、`记录一下`）；非行动句式（感受/反思/见闻，无"要做/帮我"）。

**动作**：追加到本地 `state/diary_pending_{YYYY-MM-DD}.json` → 短回复"记下了。"

`diary_pending` 单条结构：`{ "time": "HH:mm", "text": "用户原话", "source": "飞书私聊" }`

同一天连续 ≥2 条碎片 → `diary_mode=true`（`state/diary_session.json`）；用户说"拼日记/整理今天" → 立即触发 assembler。

### 8.3 diary-assembler（晚报或按需）

**输入**：当日 `diary_pending_{YYYY-MM-DD}.json`
**输出**：本地 `content/YYYY/MM/diary.md`，当日内容插入文件最顶部（日期倒序），当日节内碎片倒序。

格式：`**HH:mm**` 加粗单独一行 → 紧接原文（不改写）；每日 `---` 分隔。

完成后：清空当日 pending；质量自检：随机抽 3 条与 md 字符级一致，不通过则不写、报错。
