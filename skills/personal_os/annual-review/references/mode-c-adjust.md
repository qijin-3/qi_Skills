# Mode C：北极星调整

> 适用于：用户主动说"我想调整北极星"、"我想修改目标方向"、"我的大方向要变了"。
> **这是最高风险操作**，AI 必须保持审慎立场，不轻易执行任何修改。

---

## 守门原则（贯穿全流程）

北极星代表复利方向——修改一次就意味着过去积累的动势中断。

> AI 的立场：**不是帮用户修改，而是帮用户想清楚要不要修改。**

几个判断维度：
- 这个想法是今天突然有的，还是想了很久？
- 有没有具体事件触发（失败/受挫/新机会）？触发点会过去吗？
- 北极星修订记录里最近有没有类似的修改提议？

---

## Step 0：冲动检测

Read `{PERSONAL_OS_ROOT}/content/north-star.md` 中的「修订记录」表格。

**若发现 3 个月内已有修订记录**，先提示：

```
我看到你在 {日期} 刚调整过北极星（{当时的修订内容}），距今只有 {N} 天。

频繁修改北极星会中断复利积累。在我们继续之前，我想先确认：
这次想法跟上次有什么不同？是新的信息让你看到了更清晰的方向，还是最近有些挫折让你想换方向？
```

等用户回答，根据回答判断是否继续。

**若用户说"就是想改"但无法给出清晰理由** → 不进行修改，说：

```
我理解你有一些想法，但修改北极星是个大决定。
建议先把这个想法记到飞书任务里，放一周观察一下，如果还觉得需要改，我们再认真讨论。
```

---

## Step 1：澄清调整范围

```
你说想调整北极星，我想先了解一下：

A. 核心方向要整体调整（你觉得"这几年要做什么"本身变了）
B. 某个支柱需要调整（比如"产品杠杆"这个方向要换成别的）
C. 刻意放弃清单要更新（加一条禁区或删一条）
D. 衡量标准要修改（标准还合理，但具体数字要改）

大概是哪种？（可以组合）
```

等用户回答，确定调整范围（A/B/C/D，可组合）。

---

## Step 2：展示放弃代价

Read `{PERSONAL_OS_ROOT}/content/north-star.md`，针对用户想调整的部分，先列出"如果改，会失去什么"：

```
在我们修改之前，我想先说清楚改了会有什么代价：

{若 A 核心方向}
当前方向「{核心方向}」你已经坚持了 {N} 个月（自 {建立日期}）。
在这方向上已有的积累：{列出已完成的相关任务/目标，2-3 条有代表性的}
换方向意味着这些积累的复利效应将归零，需要从头建立新方向的认知和产出。

{若 B 某支柱}
「{支柱名称}」支柱在过去 {N} 个月：
- 月目标完成度均值：{X}%
- 完成的相关任务：{N} 条
改掉这个支柱意味着放弃这 {N} 条任务建立的积累。

{若 C 放弃清单}
「{放弃项}」是你在 {日期} 加入的，用来防止 {某类诱惑}。
删除它意味着这个防护消失了。

---

确认你理解这个代价之后，我们再继续。（说"我理解"或"我确认"）
```

**必须等待明确确认**，不能自动继续。

---

## Step 3：收集新方向的具体想法

根据调整范围，有针对性地问问题：

**若 A 核心方向变化：**
```
核心方向要变，我想先理解新的方向：

1. 新方向的核心是什么？一句话说清楚。
2. 为什么现在的时机适合转方向？
3. 转向后，你认为需要多久才能建立在新方向上的基础产出？
```

**若 B 某支柱调整：**
```
「{支柱名称}」要换成什么？
- 新支柱做什么？
- 对应清单是哪个（Writing/Projects/Growing）？
- 年底怎么判断新支柱做到了？
```

**若 C/D（范围较小）：**
直接问具体的新内容是什么。

等用户提供完整回答。

---

## Step 4：起草修订方案

基于用户回答，起草具体的修改内容（不是整体重写，是精确的差异）：

```
【修订草案】

修改类型：{A/B/C/D}

改前：
{原内容，精确引用}

改后：
{新内容}

修订记录追加：
| {今日日期} | {修订摘要} | {触发原因（用户说的）} |
```

---

## Step 5：质量门（与初始化相同四关，缩略版）

对修改后的整体北极星（不只是改动部分）做一次快速检查：

- **聚焦**：改后三支柱还能一个人做到吗？
- **可测量**：修改的衡量标准还能 30 秒判断？
- **杠杆**：至少一个支柱是媒体/产品杠杆？
- **放弃清单**：还足够有杀伤力？

有问题 → 指出，给方案，等用户打磨。

---

## Step 6：二次确认（关键安全门）

```
最终修订版本如下：

{完整的 north-star.md 变更 diff 或全文}

这将更新你的北极星文档并写入修订记录。

请说「我确认修改北极星」来完成。
```

**只接受精确短语「我确认修改北极星」**（或用户明确说"确认"/"好的就这样"等明确确认语）。

"先试试"/"我觉得可以"/"差不多"这类模糊表达 → 追问："你是确认要写入吗？请直接说确认。"

---

## Step 7：执行修改

收到明确确认后：

**7a. 更新 `{PERSONAL_OS_ROOT}/content/north-star.md`**

精确修改对应内容，在修订记录表追加一行：

```markdown
| {YYYY-MM-DD} | {修订摘要，≤20字} | {触发原因} |
```

**7b. 若北极星支柱名称或内容改变，同步更新 TABLE_NORTH_STAR**

精确更新对应支柱记录：
```bash
lark-cli base +record-upsert --as user \
  --base-token "<FEISHU_BASE_TOKEN>" \
  --table-id "<TABLE_NORTH_STAR>" \
  --record-id "<对应支柱 record_id>" \
  --json '{
    "支柱名称":"<新支柱名>",
    "核心方向":"<新说明>",
    "年底衡量标准":"<新标准>"
  }'
```

若支柱名称改变，还需检查 `TABLE_MONTHLY` 中关联该支柱的月度目标：
- 双向关联字段会自动更新显示名
- 但历史目标的完成度和状态不可改动

**7c. 输出确认摘要**

```
✅ 北极星已更新

修改内容：{修订摘要}
修订记录已写入 north-star.md
```

---

## 紧急终止条款

在以下情况，**立即停止流程，不修改任何文件**：

1. 用户在 Step 2 看完代价后说"算了"/"先不改了"
2. 用户在 Step 6 确认前说"再想想"/"先搁置"
3. 用户无法对 Step 3 的问题给出清晰回答（给 3 次机会后仍无法回答）

终止时说：

```
好的，北极星暂时不动。
你的想法我记录到了 state/north_star_draft.json，
等你想清楚了随时说"我想调整北极星"，我们再继续。
```

并将当前讨论草案写入 `{PERSONAL_OS_ROOT}/state/north_star_draft.json`（JSON格式，包含日期、讨论摘要、用户提出的新方向）。
