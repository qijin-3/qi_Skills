# AI 分析失败模式

本文档详细说明了 AI 驱动用户研究分析中的四种常见失败模式及其修复方法。基于 Caitlin Sullivan 的研究（使用 AI 测试客户发现工作流程超过 2,000 小时）。

## 失败模式 #1：虚构证据

### 表现形式

AI 生成不存在的引言：

1. **完全虚构的引言** - 在所有主要 LLM 中仍然常见
2. **"弗兰肯斯坦引言"** - 从多个来源拼接而成的引言，代表了用户所说的内容但不是他们的实际原话（在 ChatGPT 中特别常见）

如果不手动验证，这两种情况都很难被发现，但通常由提示实践导致。

### 发生原因

LLM **生成**文本而不是像搜索引擎那样**检索**文本。它们预测引言应该是什么样子，而不是实际上是什么。

- "逐字逐句"是模糊的 - 确切字符？标点符号可以不同吗？填充词呢？
- 模型用您看不到的假设填补空白
- 参与者 ID 和时间戳可能是伪造的（`[P03, 14:30]` 看起来很权威，但如果是虚构的就毫无意义）

### 解决方案：引言选择规则 + 验证

**步骤 1：定义引言规则**

将此添加到您的分析提示词中：

```
引言选择规则

从思想开始的地方开始，直到完全表达
包含推理，而不仅仅是结论
保留模糊限制词 - 它们表示不确定性
包含情绪语言（如果有）
使用参与者 ID 和大约时间戳引用 [P02 ~14:30]
不要组合面试中不同部分的陈述
如果引言超过 3 句话，将其拆分为单独的引言
```

**步骤 2：使用前验证**

在初始分析后，运行此验证提示词：

```
引言验证

对于上述分析中的每条引言：

确认引言在源记录中逐字存在
如果引言是接近的释义但不确切，标记它并提供实际措辞
如果无法找到引言，标记为未找到

输出格式：
引言：[引言]
状态：已验证 / 释义 / 未找到
如果释义：实际措辞：[他们实际说的内容]
位置：[参与者 ID，时间戳或行号]
```

## 失败模式 #2：错误或通用洞察

### 表现形式

AI 发现的主题过于宽泛或通用，无法采取行动：

- "价格是决策的一个因素"
- "人们重视可靠性"
- "用户想要更多实时信息"

这些可以描述您类别中的任何产品，并且不能指导艰难的决策。

### 发生原因

**AI 默认寻找共识：**

- LLM 是模式查找机器，呈现明显的模式
- 它们找到多个参与者提到的内容并生成模式匹配的主题
- 真正重要的洞察可能只有少数人提到，但具有重大的业务影响

**LLM 带来训练先验：**

- 如果模型看到过数千份分析，其中"价格"是第一大主题，即使您的数据不支持，它也会向价格倾斜
- 当您的提示词提到"定价问题"时，响应突然被编码为与定价相关

**调查的模糊性更糟：**

- "不适合我"可能意味着：太贵、数据量太大、不想再充电一个设备、需要屏幕
- 一个响应对产品决策有四种完全不同的含义
- AI 在没有上下文的情况下将响应归入通用主题

### 解决方案：指导解释的上下文加载

有效的上下文加载有**四个组成部分**：

1. **项目上下文** - 范围和风险
   - "探索是否添加屏幕"（具有约束的特定决策）
   - 不是"做用户研究"（模糊，导致通用分析）

2. **业务目标** - 您试图实现的目标
   - "需要知道功能是否会吸引新用户而不是疏远现有用户"
   - AI 权重证据以回答您的实际问题

3. **产品上下文** - 领域知识
   - 没有："我想看我的数据"是通用的
   - 有：AI 在无屏幕可穿戴设备与 Apple Watch 的背景下理解这一点

4. **参与者概述** - 谁在说话
   - "我需要实时数据"来自已流失的 Garmin 切换者的含义与来自忠实用户的含义不同
   - AI 只有知道证据来自谁才能正确权重证据

**对于面试**，在分析提示词中构建上下文（参见 [prompt-templates.md](prompt-templates.md)）。

**对于调查**，始终指定：
- 哪些列包含客户声音与元数据
- 模糊术语的含义
- 您试图做出的决策

## 失败模式 #3：无法指导决策的"信号"

### 表现形式

AI 在没有足够证据或细微差别的情况下生成自信的推荐：

- 樱桃挑选三条热情的引言并跳转到"是的，构建这个"
- 忽略矛盾的证据
- 将表面级请求视为字面功能要求
- 不标记风险或边缘情况

### 发生原因

- AI 被训练为乐于助人和自信，而不是怀疑
- 它优先考虑寻找模式而不是质疑它们
- 如果没有明确指导，它会将复杂性扁平化为整洁的摘要
- 它不会自动权重证据质量或标记假设

### 解决方案：要求证据锚定 + 风险标记

**添加到您的提示词中：**

```
分析要求

1. 证据锚定：
- 对于每个洞察，提供具体的参与者引用
- 注意提到此内容的参与者比例（例如，"3/10 参与者提到"）
- 当证据薄弱时标记（例如，"基于单次提及，可能不具有代表性"）

2. 矛盾检测：
- 明确寻找和报告矛盾的陈述
- 注意参与者在面试中何时改变主意
- 报告人们所说的与他们行为之间的张力

3. 风险标记：
- 识别分析中的假设
- 标记与主要主题相矛盾的证据
- 注意可能重要的边缘情况或异常值
- 如果置信度低，请说明

4. 细微差别优于简化：
- 不要过度简化复杂的模式
- 存在混乱时报告混乱
- 区分"一些用户想要 X"与"大多数用户想要 X"
```

## 失败模式 #4：矛盾的洞察

### 表现形式

- AI 遗漏面试内部或面试之间的矛盾
- 将主题报告为一致，而实际上存在争议
- 未能捕捉参与者的自相矛盾
- 不区分不同用户细分想要的东西

### 发生原因

- AI 旨在寻找模式，而不是异常
- 它平滑处理不一致以创建连贯的叙述
- 如果没有明确指示，它不会寻找矛盾
- 它不会自动按不同需求细分用户

### 解决方案：显式矛盾检查 + 用户细分

**添加到您的提示词中：**

```
矛盾检查

1. 面试内矛盾：
- 注意参与者何时自相矛盾或反驳自己
- 报告语气转变或重新框架（例如，"在第 8 分钟他们说 X，但在第 35 分钟他们将其重新框架为 Y"）
- 标记热情或确定性何时变化

2. 面试间矛盾：
- 明确寻找和报告不同参与者说的相反内容
- 注意分歧模式
- 不要平滑处理这些 - 将它们报告为张力

3. 用户细分：
- 不同的用户可能想要不同的东西 - 细分它们
- 当它们不同时，按细分报告洞察
- 注意主题何时适用于某些用户而不适用于其他用户
- 区分当前用户、已流失用户和潜在用户
```

## 关键原则

1. **验证是不可协商的** - 始终验证引言和证据
2. **上下文很重要** - 提供项目、业务、产品和参与者上下文
3. **拥抱混乱** - 真实研究是混乱的；不要过度简化
4. **标记不确定性** - 报告置信度水平和弱证据
5. **寻找矛盾** - 它们通常比共识更有价值
6. **细分用户** - 不同的用户想要不同的东西

## 何时使用此参考

在以下情况下阅读本文档：
- 诊断为什么 AI 分析感觉不对或不可信
- 为面试或调查设计分析提示词
- 审查 AI 生成的洞察的准确性
- 培训团队可信赖的 AI 分析实践
